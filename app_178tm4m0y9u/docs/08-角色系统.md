## 1. 系统总览

角色系统是玩家身份的承载者，与宠物系统并列为本游戏两大核心模块。玩家角色包含 **等级、装备、强化、转生、天赋、成就、战令、日常任务** 等多个子系统，所有属性均通过 `G.player` 全局状态对象维护。

## 2. 角色等级系统

### 2.1 经验曲线公式

升级所需经验由 `getExpForLevel(lv)` 计算（`systems.js:266`）：

```js
function getExpForLevel(lv) {
  return Math.floor(100 * Math.pow(lv, 1.6) + 100);
}
```

| 等级 | 升下一级所需经验 | 累计粗略值 |
|---|---|---|
| Lv 1 → 2 | 200 | 200 |
| Lv 10 → 11 | ≈ 2,190 | ≈ 8,500 |
| Lv 50 → 51 | ≈ 25,700 | ≈ 350,000 |
| Lv 100 → 101 | ≈ 129,000 | ≈ 1,700,000 |

> **平衡备注**（代码注释）：早期版本为 `1.8` 倍增长，后调为 `1.6`，缓解后期升级过慢。

### 2.2 加经验函数 `addExp(amount)`

定义于 `systems.js:273`，是所有经验获取的统一入口：

1. 计算 buff 与天赋加成后的实际经验倍率：
   ```
   mult = getBuffMult('exp_mult')
        × (1 + getTalentBonus('exp_bonus')
              + getTalentBonus('exp_mastery')
              + getTalentBonus('growth_mastery'))
   ```
2. 累加经验 `G.player.exp += Math.floor(amount * mult)`。
3. 循环升级：当经验达到 `expToNext` 且 `level < maxLevel` 时：
   - 扣除经验、等级 +1、刷新 `expToNext`；
   - **天赋点发放规则**（关键）：
     - 阈值 `talentThreshold = rebirth × 10`；
     - 仅当 `level > talentThreshold` 且 `level > talentPointsEarned` 时发放 1 点；
     - 同时更新 `talentPointsEarned = level`；
   - 同步所有宠物等级：`G.pets.forEach(p => p.level = G.player.level)`。

> **天赋点不重复发放**：转生后已获得天赋点的等级不会再发，只有突破新等级上限才补发新点数（详见 §3）。

### 2.3 等级对属性的影响

人物基础属性随等级线性增长（`ui.js:3434-3435`、`ui.js:651-657`）：

| 属性 | 基础公式 |
|---|---|
| 力量 | `10 + lv × 3` |
| 体质 | `10 + lv × 2` |
| 敏捷 | `10 + lv × 2` |
| 耐力 | `10 + lv × 2` |
| 魔力 | `10 + lv × 2` |
| 气血 | `50 + lv × 10` |

> **v2.6.0 变更**：人物属性面板由四维升级为五维——新增"耐力"属性，原"智力"重命名为"魔力"。详见 [§14](#14-v260-更新五维属性体系)。

> 战斗中怪物的经验/金币奖励会随 `lvDiff`（怪物等级 - 玩家等级）线性调整，但不会低于 5。

## 3. 转生系统

转生是角色成长的核心循环机制：满级后重置等级，换取永久战力加成与更高等级上限。所有逻辑位于 `economy.js:1068-1122`。

### 3.1 转生条件

```js
function canRebirth() { return G.player.level >= G.player.maxLevel; }
```

- 当前等级必须达到 `maxLevel`（初始 100，每转生 +10）。
- 转生入口：导航栏 `🔄 转生` 页面，或 `renderRebirthScreen()`（`ui.js:4483`）。

### 3.2 转生流程 `doRebirth()`

| 步骤 | 操作 |
|---|---|
| 1 | 校验 `canRebirth()`，不满足返回 false |
| 2 | 保留已发放天赋点的最高等级：`G.talentPointsEarned = Math.max(原值, G.player.level)` |
| 3 | `G.player.rebirth++` |
| 4 | `G.player.maxLevel += 10` |
| 5 | 等级重置：`level=1`、`exp=0`、`expToNext=getExpForLevel(1)` |
| 6 | 所有宠物等级同步为 1：`G.pets.forEach(p => p.level = 1)` |
| 7 | **自动卸下所有装备**（详见 §3.4） |
| 8 | 退还宝石、重置强化等级（详见 §3.4） |
| 9 | `setAchievement('rebirth', G.player.rebirth)` |
| 10 | 提示 + `saveGame()` |

### 3.3 转生奖励（属性加成）

转生奖励以**乘数形式**直接作用于战斗计算，永久有效：

| 加成对象 | 公式 | 代码位置 | 说明 |
|---|---|---|---|
| 玩家综合战力 | `× (1 + rebirth × 0.15)` | `battle.js:12` | 每转生 +15% 战力 |
| 战斗经验获取 | `× (1 + rebirth × 0.20)` | `battle.js:260` | 每转生 +20% 经验 |
| 战斗金币获取 | `× (1 + rebirth × 0.10)` | `battle.js:264` | 每转生 +10% 金币 |
| 宠物攻击力 | `× (1 + rebirth × 0.10)` | `battle.js:618` | 每转生 +10% 宠物攻击 |
| 宠物灵力 | `× (1 + rebirth × 0.10)` | `battle.js:646` | 每转生 +10% 宠物灵力 |
| 爬塔最大层数 | `100 + rebirth × 50` | `config.js:1726` | 每转生 +50 层 |
| 装备强化上限 | `12 + 转生加成`（见 §10.1） | `ui.js:7337` | 每转生 +1，每 5 次再 +1 |

> **注意**：转生奖励**不是**通过 `REBIRTH_BONUSES` 常量统一管理，而是分散在战斗、爬塔、强化等多处代码中按 `G.player.rebirth` 直接计算。

### 3.4 转生时装备与强化处理

转生会自动**卸下所有装备**回背包，并重置槽位强化等级：

```js
Object.keys(G.player.equipment).forEach(function(slotId) {
  var item = G.player.equipment[slotId];
  if (item) {
    // 1. 退还宝石孔中的宝石到宝石背包
    if (Array.isArray(item.gemSlots)) {
      item.gemSlots.forEach(function(gslot) {
        if (gslot.gem && gslot.gem.level > 0) {
          addGemToBag(gslot.gem.type, gslot.gem.level, 1);
          gslot.gem = null;
        }
      });
    }
    // 2. 装备本体回背包
    G.equipmentBag.push(item);
    G.player.equipment[slotId] = null;
  }
  // 3. 旧 G.gems 残留数据退还
  // 4. 重置槽位强化等级（绑定槽位而非装备）
  if (G.player.forgeLevels && G.player.forgeLevels[slotId]) {
    G.player.forgeLevels[slotId] = 0;
  }
});
```

| 处理项 | 行为 |
|---|---|
| 装备本体 | 全部退还到 `G.equipmentBag` |
| 镶嵌宝石 | 全部退还到 `G.gemBag` |
| 槽位强化等级 | **清零**（强化绑定槽位，不绑定装备） |
| 旧版 `G.gems` | 兼容退还 |

### 3.5 宠物品质重生后重新计算

转生本身**不直接修改宠物品质**，但游戏在加载存档时会调用 `migratePetRarity()`（`state.js:291`），通过 `recalcRarity(pet)`（`pet.js:148`）根据宠物当前成长/资质重新计算品质：

```js
function recalcRarity(pet, qualityBoost) {
  if (pet.rarity === 'special') return 'special'; // 异化品质固定
  // 综合：成长得分 × 0.45 + 资质得分 × 0.55
  var qualityScore = gScore * 0.45 + aptScore * 0.55;
  if (qualityScore < 0.40) return 'common';
  if (qualityScore < 0.65) return 'uncommon';
  if (qualityScore < 0.85) return 'rare';
  if (qualityScore < 0.95) return 'epic';
  if (qualityScore < 0.99) return 'legendary';
  return 'mythic';
}
```

> 该机制同样适用于：宠物融合、月华露培养、归元丹/归虚丹重生等场景——这些操作改变成长/资质后，下次 `recalcRarity` 调用时品质自动刷新。

### 3.6 转生后天赋点发放规则

为避免转生后玩家从 1 级重新升级时再次获得已领过的天赋点：

| 转生次数 | 等级上限 | 新增可发天赋点的等级区间 | 新增点数 |
|---|---|---|---|
| 0 转 | 1~100 | 1~100 | 100 点 |
| 1 转 | 101~110 | 101~110 | 10 点 |
| 2 转 | 111~120 | 111~120 | 10 点 |
| ... | ... | ... | ... |
| 6 转（满） | 151~160 | 151~160 | 10 点 |

- 通过 `G.talentPointsEarned` 记录"已发放天赋点的最高等级"。
- 转生时 `talentPointsEarned = max(原值, 当前等级)`，避免重置。
- 升级时仅在 `level > rebirth × 10` 且 `level > talentPointsEarned` 时才发放 1 点。
- **v2.9.0 上限**：`TALENT_POINT_CAP = 160`，6 转累计 160 点（0 转 100 点 + 1~6 转各 10 点）。
- 天赋树总点数 320 点，玩家仅能获得 160 点，**无法点满全部天赋**（详见 §4）。

## 4. 天赋系统（星图）

天赋系统定义于 `systems.js:141-308`，UI 渲染于 `ui.js:3117-3270`。

> **v2.9.0 重构（需求4.2）**：天赋系统全面重构为**暗黑类分支天赋树**架构。采用「分支节点+前置解锁路径」，划分输出/生存/资源/辅助 4 大专精方向；节点分普通/大天赋/核心三级；玩家可获取天赋点总数固定 160 点，整棵天赋树需 320 点才能点满，**无法点满全部天赋**。

### 4.1 天赋点数获取

- **唯一来源**：角色升级（详见 §2.2）。
- **不重复发放**：转生后已发放等级不再发放。
- **v2.9.0 上限**：`TALENT_POINT_CAP = 160`（6 次转生达 160 级即可获得全部 160 点）
- 状态字段：`G.talentPoints`（可用点数）、`G.talentPointsEarned`（已发放最高等级）。
- 函数：
  - `getTalentPoints()` 返回 `G.talentPoints`；
  - `addTalentPoints(n)` 增加 n 点（用于兑换码等特殊渠道）。

### 4.2 天赋树结构（v2.9.0 重构）

`TALENT_TREE` 包含 1 个中心起点 + 4 大专精分支，**总计 320 点可投**（玩家最多获得 160 点，无法点满）：

| 分支 | branch ID | 颜色 | 节点数 | 总点数 | 主题 |
|---|---|---|---|---|---|
| 输出星轨 | `output` | `#ef4444` 红色 | 18 | 80 | 攻击/暴击/技能/吸血 |
| 生存星轨 | `survival` | `#3b82f6` 蓝色 | 19 | 80 | 气血/防御/闪避/减伤 |
| 资源星轨 | `resource` | `#f59e0b` 金色 | 19 | 80 | 掉落/品质/金币/钻石 |
| 辅助星轨 | `support` | `#22c55e` 绿色 | 19 | 80 | 经验/孵化/技能/buff |
| **合计** | — | — | **75** | **320** | 满树需 320 点 |

> 玩家最多获得 160 点（6 次转生达 160 级），仅能点满 50% 的天赋树，必须取舍选择专精方向。

节点类型说明（三级分类）：

| 类型 | 圆点半径 | 含义 |
|---|---|---|
| `origin` | 26 | 星图中心，自动点亮，无效果 |
| `small` | 20 | 普通天赋，基础加成，多点可叠 |

### 4.3 天赋解锁规则

```js
function canUnlockTalent(id) {
  var def = TALENT_TREE.find(t => t.id === id);
  if (def.type === 'origin') return true;
  return def.requires.every(reqId => getTalentLevel(reqId) >= 1);
}

function upgradeTalent(id) {
  // 检查：定义存在、非 origin、未满级、前置满足、点数 ≥ 1
  G.talents[id] = lv + 1;
  G.talentPoints -= 1;
  saveGame(); render();
}
```

- 前置条件：所有 `requires` 列出的节点至少点亮 1 级。
- 每个节点可升级至 `maxLevel`，每级消耗 1 点天赋点。
- 加成公式：`getTalentBonus(id) = getTalentLevel(id) × def.perLevel`。

#### 1. 任务类型与奖励绑定

| 任务类型 | 任务目标 | 奖励 |

#### 2. 轮次循环规则

- 以玩家完成任务的自然次数计数，每 **10 次**为一轮
- 状态字段：`G.mainQuestCycleCount`（迁移于 `state.js` ~line 293-295）
- 第 **1-9 次**任务：从「经验任务、金币任务」中随机抽取任务类型与目标
- 第 **10 次**任务：**固定为钻石任务**
- 完成第 10 次任务后，计数清零重置，重新从第 1 次开始下一轮循环

```js
// economy.js → generateRandomMainQuest()
function generateRandomMainQuest() {
  G.mainQuestCycleCount = (G.mainQuestCycleCount || 0) + 1;
  if (G.mainQuestCycleCount >= 10) {
    // 第 10 次：固定钻石任务
    G.mainQuestCycleCount = 0; // 清零重置
    return generateDiamondQuest();
  } else {
    // 第 1-9 次：随机经验/金币任务
    return randomPick([generateExpQuest(), generateGoldQuest()]);
  }
}
```

#### 3. 校验标准

- 前 9 次仅出现经验/金币任务
- 第 10 次必为钻石任务
- 完成 10 次后任务计数重置，循环逻辑正确

#### 3. Buff 系统架构

| 函数 | 用途 |
|:---|:---|
| `activateBuff(buffType, mult, durationMin)` | 激活 buff，存入 `G.buffs[buffType] = { mult, expireAt }` |
| `getBuffMult(buffType)` | 获取 buff 倍率，返回 1 表示无 buff 或已过期 |
| `BUFF_DISPLAY` | UI 显示配置（图标、名称、剩余时间） |

#### 4. 各 buff 的应用逻辑

##### 4.1 全属性 buff（all_stat）

`ui.js → getCharacterBonusForPet` 中将 `all_stat` buff 作为 flat 值加成到四维属性和气血：

```js
var allStatBuff = getBuffMult('all_stat');
if (allStatBuff > 1) {  // all_stat 的 mult 即为加成数值（如 50）
  bonus.力量 += allStatBuff;
  bonus.体质 += allStatBuff;
  bonus.敏捷 += allStatBuff;
  bonus.智力 += allStatBuff;
  bonus.气血 += allStatBuff * 10;  // 气血按体质×10放大
}
```

人物属性显示（`renderCharacterScreen`）的 attrHtml 会展示 buff 加成，玩家可直观看到属性提升。

##### 4.2 双倍经验 buff（exp_mult）

`systems.js` 中经验加成倍率计算：

```js
expMult = getBuffMult('exp_mult') × (1 + exp_bonus + exp_mastery + growth_mastery)
```

最终经验 = 战斗基础经验 × `expMult`，2 小时内经验收益翻倍。

##### 4.3 双倍金币 buff（gold_mult）

`battle.js → giveMonsterKillRewards` 中金币加成：

```js
goldMult = getBuffMult('gold_mult') × (1 + gold_bonus + gold_mastery + loot_mastery)
```

最终金币 = 战斗基础金币 × `goldMult`，2 小时内金币收益翻倍。

##### 4.4 孵化速度 buff（hatch_mult）

`ui.js` 中孵化 tick 逻辑：

```js
// 每 tick 增加的进度受 hatch_mult 倍率影响
var hatchMult = getBuffMult('hatch_mult');
egg.progress += hatchMult;  // 10倍速度下每 tick 进度加快10倍
```

22 小时内孵化速度提升 10 倍，玩家可快速孵化大量宠物蛋。

##### 4.5 双倍蛋掉落 buff（egg_drop_mult）

`battle.js → openChestDirect` 中开宝箱获得宠物蛋时：

```js
var eggMult = getBuffMult('egg_drop_mult');
if (eggMult > 1 && reward.type === 'egg') {
  // 按倍率额外掉落蛋
  for (var i = 1; i < eggMult; i++) {
    G.eggs.push(generateEgg(...));
  }
}
```

2 小时内开宝箱获得的宠物蛋数量翻倍。

> v2.7.0 **未改动**角色与宠物之间的属性转移逻辑，`getCharacterBonusForPet`、`getCharacterStats` 等函数保持 v2.6.0 行为。

#### 2. 战力评价体系变化的影响

虽然属性转移链路不变，但宠物战力数值的计算口径发生变化，间接影响以下角色相关模块：

| 模块 | 影响方式 | 说明 |
|---|---|---|
| 角色综合战力 | 间接受影响 | 角色综合战力 = 角色属性战力 + 宠物战力加成；宠物战力数值变化会反映到角色综合战力显示上 |
| 出战推荐 | 直接影响 | 宠物列表按新战力排序，影响角色出战时的宠物推荐顺序 |
| 派遣奇遇 | 直接影响 | `getDispatchTotalPower` 汇总派遣宠物新战力，影响奖励加成计算 |
| 竞技场匹配 | 直接影响 | 按新战力段位匹配对手 |

#### 3. 二级属性的角色加成来源

v2.7.0 新增的 10 项二级战斗属性中，部分属性受角色装备/天赋加成：

| 二级属性 | 角色加成来源 | 说明 |
|---|---|---|
| 暴击率 | 天赋 `pet_crit`（每级 +1.5%） | 战宠天赋星轨直接加成宠物暴击率 |
| 暴击伤害 | 天赋 `pet_crit_dmg`（每级 +4%） | 暴击精通天赋 |
| 闪避率 | 天赋 `pet_dodge`（每级 +1.5%） | 灵动之躯天赋 |
| 吸血 | 天赋 `pet_lifesteal`（每级 +2%） | 吸血之印天赋 |
| 减伤 | 天赋 `pet_resolve`（每级 +3%） | 坚毅之心天赋 |
| 技能触发 | 天赋 `pet_skill_trigger`（每级 +2%） | 法术共鸣天赋 |
| 技能增伤 | 天赋 `pet_skill_dmg`（每级 +3%） | 法术穿透天赋 |

- 这些天赋加成原本就存在于 v2.6.0 战宠星轨中（见 [§4.4.3](#443-战斗星轨combat)），v2.7.0 将其纳入二级属性战力计算后，**点亮相关天赋的宠物战力会相应提升**；
- 角色装备词条（`crit_rate`/`dodge_rate`）通过 `getCharacterBonusForPet` 间接影响宠物二级属性，同样纳入新战力计算。

#### 4. 寿命系数的影响

- 寿命系数仅作用于**宠物战力评分**，不影响角色属性与角色-宠物属性转移；
- 角色综合战力中的宠物加成部分会因宠物寿命降低而折减，鼓励玩家维持宠物寿命。

#### 5. 变更总结

| 维度 | v2.6.0 | v2.7.0 |
|---|---|---|
| 属性转移链路 | 五维 20% 加成 | **不变** |
| 宠物战力构成 | 五维派生属性战力 × 技能倍率 | 五维派生属性战力 + 二级属性战力 × 技能倍率 × 寿命系数 |

## v2.8.0 更新

> **版本**：v2.8.0 轮回塔转生按钮置灰修复
> **更新日期**：2026-07-12

### 一、挑战失败后转生按钮置灰修复（需求5.3）

v2.8.0 修复轮回塔挑战失败结算界面中转生按钮被异常置灰、无法执行转生操作的 Bug。本修复涉及 `ui.js` 中轮回塔 UI 渲染分支逻辑。

#### 1. 原 Bug 根因

轮回塔 UI 按挑战状态分为两个分支：

| 分支 | 触发条件 | 旧逻辑（v2.7.0） |
|---|---|---|
| 「挑战中」分支 | `samsara.inChallenge === true` | 包含「结算转生」按钮（满足 `canSettle` 时可点击，否则置灰） |
| 「未在挑战中」分支 | `samsara.inChallenge === false` | **无「结算转生」按钮**，仅显示「开始挑战」按钮 |

`samsaraFailChallenge()`（`economy.js`）在挑战失败时执行：

```js
function samsaraFailChallenge() {
  if (!G.samsara) return;
  G.samsara.inChallenge = false;  // 切换到「未在挑战中」分支
  saveGame();
}
```

**Bug 链路**：
1. 玩家挑战失败 → `samsaraFailChallenge()` 将 `inChallenge` 置为 `false`
2. UI 切换到「未在挑战中」分支
3. 「未在挑战中」分支无「结算转生」按钮 → 等同按钮被置灰
4. 即使玩家已通关 ≥10 层且达到满级（满足 `canSettle` 条件），也无法点击转生

#### 2. `canSettle` 条件定义

```js
// ui.js
var canSettle = samsara.currentFloor >= SAMSARA_REBIRTH_MIN_FLOOR
              && G.player.level >= G.player.maxLevel;
```

- `SAMSARA_REBIRTH_MIN_FLOOR`：结算转生最低通关层数（默认 10）
- `G.player.level >= G.player.maxLevel`：玩家达到等级上限（满级）
- 两个条件均满足时 `canSettle = true`

#### 3. 修复方案

在「未在挑战中」分支的按钮区追加「结算转生」按钮，仅当 `canSettle === true` 时显示：

```js
// ui.js 「未在挑战中」分支
'<div class="flex gap-2 flex-wrap">' +
  '<button class="btn-primary px-6 py-3" onclick="startSamsaraUI()">🌀 开始挑战</button>' +
  // v2.8.0 需求5.3：未在挑战中时，若已满足结算条件，也展示可点击的结算转生按钮
  (canSettle ?
    '<button class="btn-gold px-6 py-3" onclick="confirmSamsaraSettle()">✨ 结算转生</button>' : '') +
'</div>' +
(samsara.maxFloorCleared > 0 ? '<p class="text-xs text-secondary mt-2">上次最高通关：' + samsara.maxFloorCleared + '层</p>' : '') +
(canSettle ? '<p class="text-xs text-yellow-400 mt-2">✅ 当前可结算转生！将获得 <span class="font-bold">' + (typeof calcSamsaraPoints === 'function' ? calcSamsaraPoints(samsara.currentFloor) : 0) + ' 轮回积分</span></p>' :
  (samsara.currentFloor >= SAMSARA_REBIRTH_MIN_FLOOR ? '<p class="text-xs text-green-400 mt-2">✅ 已通关' + SAMSARA_REBIRTH_MIN_FLOOR + '层，达到转生最低门槛。继续挑战可获得更多积分，或达到满级后结算转生。</p>' : ''))
```

#### 4. 修复后行为对比

| 场景 | 旧逻辑（v2.7.0） | 新逻辑（v2.8.0） |
|---|---|---|

#### 掉落收益分支拆分

原「资源」分支拆分为 3 个子分支：

## v2.11.0 更新

### 天赋第二套加点方案（需求6.1）
- **双方案存储**：新增`G.talents2`（第二套天赋方案）、`G.activeTalentBuild`（当前激活方案1/2）、`G.talentBuild2Unlocked`（是否已解锁）、`G.totalTalentPointsEarned`（总获得点数，双方案共用）字段。
- **核心函数重构**：`getTalentLevel(id)`从当前激活方案读取；`getAllocatedTalentPoints(build)`计算指定方案已分配点数（origin免费不计入）；`getTalentPoints()`改为`totalEarned - allocated`；`upgradeTalent(id)`修改当前激活方案。
- **切换/解锁**：`switchTalentBuild(build)`切换方案（战斗中禁止切换）；`unlockTalentBuild2()`消耗500钻石解锁第二套方案。
- **存档迁移**：旧存档从`talentPoints`（剩余池）+当前方案已分配点数推算`totalTalentPointsEarned`，自动初始化新字段。
- **天赋点发放**：`addTalentPoints(n)`改为写入`G.totalTalentPointsEarned`（上限160），`addExp`中升级获得天赋点也改为调用`addTalentPoints(1)`。

### 属性点系统重制（需求3.1/3.2/3.3）
- **双层模型**：宠物属性点改为双层模型——`pet.attrPoints`包含固定加成（每级每维+1）+自由分配点；`pet.freeAttrPoints`为剩余未分配自由点。
- **转生后属性重置**：转生后所有宠物等级重置为1级，固定加成属性重置为1级初始值，自由属性点全额返还。
